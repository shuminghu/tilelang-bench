# TileLang Coding-Agent Benchmark — 100-Task Design

Source repo: **`tile-ai/tilelang`** (DSL for high-performance GPU kernels; >1k merged PRs).
All tasks are scored automatically to a scalar in **[0, 1]** and run in the isolated,
no-Docker harness under `harness/` (golden venv + disposable per-task workspace, grader
kept private, network/credentials scrubbed, history-stripped sources — see anti-cheat).

## Why this repo / why these tasks
TileLang is a *performance* DSL: a kernel can be correct yet 10x too slow, so the
benchmark measures both **correctness** and **speed** against the repo's own tuned
kernels. That makes it hard to saturate (a model must actually understand tiling,
pipelining, memory layout, and the hardware), and the repo ships verified tuned
kernels + regression tests we can reuse as ground truth.

## Tracks (100 tasks total — as built & GPU-validated)

| Track | Count | Agent does | Score in [0,1] | Scorer |
|------|------|-----------|----------------|--------|
| **P — Perf optimize** | 30 | Make a correct-but-slow kernel fast | `clamp01((log t_base − log t_agent)/(log t_base − log t_target))`, hard correctness gate | `score_perf.py` |
| **C — Implement**     | 56 | Write a kernel from a torch spec (stub start) | fraction of (shape) cases passing `allclose` (+ must be a real TileLang kernel) | `score_correct.py` |
| **D — Debug**         | 14 | Fix a planted bug in a real TileLang kernel | fraction of cases the fix makes correct | `score_correct.py` |

Every task is generated by `harness/gen_tasks.py`, validated on GPU by
`harness/validate_tasks.py` (anchors/oracles compile + are correct; perf targets are
measurably faster than baselines), and the verified set is composed into exactly 100 by
`harness/select_tasks.py` → `tasks/manifest.json`. Validation summary: perf 36/36 pass
(median target/baseline speedup **5.4×**, range 1.25–6.3×), implement 59/67, debug 14/14.

### P — Performance optimization (45)
- **Premise:** `solution.py::build(...)` returns a *working but detuned* kernel. Optimize toward the repo's tuned configuration.
- **Anchors:** `baseline` (detuned → score 0.0) and `target` (repo-tuned → score 1.0) live in the **private** `problem.py`. Partial credit by log-latency interpolation; >target ⇒ clamped to 1.0.
- **Timing robustness (no clock pinning):** anchors and candidate are measured **interleaved** each round so drift is common-mode and cancels in the ratio; warmups + median over rounds + relative-stdev gate with re-measure. Validated empirically (~0.3% ratio rel-std on H100 without `-lgc`).
- **Generation:** verified template library × a grid of {problem shape, dtype, detune gap}. Families: `gemm` (fp16/bf16), `gemm+bias` epilogue, `gemv`, fused `elementwise`, `rmsnorm`, `layernorm`, row `softmax`, `transpose/swizzle`, `dequant_gemm` (int4→fp16), `flash_attention` tile.

### C — Implement from spec (40)
- **Premise:** `solution.py` is a stub with the signature + docstring spec; the agent writes the TileLang kernel. The torch reference + the (shape,dtype) case list are in the private `problem.py`.
- **Score = (#cases correct)/(#cases)** → smooth partial credit (e.g. gets fp16 right but fails bf16 / odd shapes / non-multiple-of-tile sizes). Optional tiny perf floor to reject trivial torch-passthrough is enforced by requiring the output be produced by a compiled TileLang kernel (grader imports `build` and checks it returns a tilelang kernel object).
- **Generation:** same family library as P, but starter is a stub and the case grid emphasizes correctness edge cases (ragged shapes, dtype variety, broadcasting).

### D — Debug planted bug (14)
- **Premise:** the agent gets a real TileLang kernel (gemm / pointwise) that **compiles but is wrong** — a planted defect of the kind that occurs in practice (missing accumulator `T.clear`, wrong operator, dropped nonlinearity). The agent must locate and fix it.
- **Score = (#cases correct)/(#cases)** via `score_correct.py`. Contamination-resistant: the bug is ours, not in git history, so it can't be looked up.
- **Why not historical PRs (yet):** the most authentic variant — checking out the parent of a real bug-fix commit and making its `testing/...` test pass — needs an *editable per-task build* of TileLang (the fix lives in package source, not the workspace), i.e. `snapshot_repo.sh` + per-task venv/worktree + `ccache`. That scaffolding exists (`snapshot_repo.sh`) but a full rebuild per task is costly on a shared box; the planted-bug track gives the same "debug real kernel code" signal automatically and cheaply. Historical-PR regression is the designed next extension (see Shortcomings).

## Anti-cheat (applies to all tracks)
- Grader files (`problem.py`, `task.json`, reference fix, tuned anchors) stay in the **private** `TASK_DIR`; only `solution.py` + `prompt.txt` are copied into the agent's cwd.
- Trajectory written **outside** the agent's cwd.
- Sources are **history-stripped** (`git archive`) so the upstream PR/solution isn't reachable.
- Agent subprocess runs **offline** (dead proxy, `PIP_NO_INDEX`, `UV_OFFLINE`, `GIT_ALLOW_PROTOCOL=file`) with **credentials blanked**, so it can't fetch the fix or exfiltrate keys.

## Execution
- One GPU per run via `HARNESS_GPU`; parallel fan-out across worker-4's 8 GPUs (`run_sweep_parallel.py`), shared NFS.
- Each task: `setup_task.sh` → agent run (`run_agent.py`, mini-swe-agent) → `score_*.sh` with the private grader → `reset_task.sh`.

## Task manifest
Generated under `tasks/<track>/<family>__<config>/` by `harness/gen_tasks.py`; the full
list with ids/configs is emitted to `tasks/manifest.json`.
