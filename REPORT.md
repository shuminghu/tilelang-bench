## 1. Summary
An unsaturated, automatically-scored benchmark for coding agents built on
**`tile-ai/tilelang`** (a GPU-kernel DSL with >1k merged PRs). 100 tasks across three
tracks exercise the full kernel-engineering loop ‚Äî **writing** kernels, **optimizing**
them for speed, and **debugging** them ‚Äî each scored to a scalar in `[0, 1]`. Three
models are benchmarked with `mini-swe-agent` over an OpenAI-compatible gateway.

## 2. Why TileLang
- **Performance is the point.** A kernel can be correct yet 10√ó too slow, so the
  benchmark grades *speed against the repo's own tuned kernels*, not just correctness ‚Äî
  hard to saturate without real understanding of tiling, pipelining, and memory layout.
- **Ground truth ships with the repo.** 100+ tuned example kernels and 248 test files
  give verifiable correctness + performance anchors.
- **Single-device friendly.** Kernels compile/run on one GPU (we used one H100),
  unlike multi-node training repos.

## 3. Environment
- **No Docker** (unprivileged host, no `docker` group / user-namespaces). Isolation via:
  - **Golden venv** (`.venv`, read-only shared deps incl. `tilelang==0.1.11`,
    `apache-tvm-ffi==0.1.11`, torch+CUDA).
  - **Disposable per-task workspace** (`setup_task.sh`): only `solution.py` + `prompt.txt`
    are agent-visible; all caches (`HOME`,`TMPDIR`,`*_CACHE_DIR`,‚Ä¶) redirect into the
    workspace so reset is `rm -rf`.
  - **Driver venv** (`.venv-agent`) runs `mini-swe-agent`; the agent's shell commands run
    in a `LocalEnvironment` pinned to one GPU via `HARNESS_GPU`/`CUDA_VISIBLE_DEVICES`.
- **Anti-cheat:** private grader (`problem.py`/anchors/oracle never enter the workspace);
  trajectory written outside cwd; sources history-stripped (`git archive`); agent
  subprocess forced **offline** (dead proxy, `PIP_NO_INDEX`, `UV_OFFLINE`,
  `GIT_ALLOW_PROTOCOL=file`) with **credentials blanked** so it can't fetch the upstream
  fix or exfiltrate keys.
- **Execution:** one GPU per run, fanned out across worker-4's 8 GPUs over shared NFS
  with `run_sweep_parallel.py`.
- **Timing robustness (no clock pinning):** baseline/candidate/target measured
  **interleaved** each round (drift is common-mode and cancels in the ratio) + warmups +
  median over rounds + rel-stdev gate with re-measure. Empirically ~0.3% ratio rel-std
  on H100 without `nvidia-smi -lgc`.

## 4. Tasks (100)
Generated by `harness/gen_tasks.py` from a GPU-verified kernel-template library,
validated by `harness/validate_tasks.py`, composed to exactly 100 by `select_tasks.py`
(`tasks/manifest.json`). See `tasks/DESIGN.md` for the full taxonomy.

| Track | Count | Agent does | Score |
|------|------|-----------|-------|
| **Perf** | 30 | Optimize a slow GEMM (fp16/bf16, varied shapes) toward the repo-tuned config | log-speedup ratio vs anchors, correctness-gated |
| **Implement** | 56 | Write a kernel from a torch spec (pointwise/fused/bias/transpose/cast/gemv/gemm) | fraction of shape cases passing `allclose` |
| **Debug** | 14 | Fix a planted bug in a real TileLang kernel | fraction of cases the fix corrects |

**Task validation:** perf 36/36 candidates pass (median target/baseline speedup **5.4√ó**,
range 1.25‚Äì6.3√ó); implement 59/67; debug 14/14. Dropped: 8 reduction kernels (register
overflow) and gemv-perf (no clean speed knob) ‚Äî see Shortcomings.

## 5. Scoring
- **Perf** (`score_perf.py`): `score = clamp01((log t_base ‚àí log t_agent)/(log t_base ‚àí log t_target))`,
  with a hard correctness gate (`allclose`, per-task `rtol/atol`). 0 = baseline, 1 = repo-tuned, >tuned clamps to 1.
- **Implement / Debug** (`score_correct.py`): `score = (#cases correct)/(#cases)`; the
  returned object must be a compiled TileLang kernel (blocks torch-passthrough cheats).
- Dispatch by `track` in `score_task.sh`. Smoke-tested: a debug task scores **0.0 buggy ‚Üí 1.0 fixed**.

## 6. Models & harness
- Driver: `mini-swe-agent` (`DefaultAgent`, native tool-calls) via `litellm` ‚Üí
  **OpenCode Zen** gateway. `CleanLitellmModel` strips `provider_specific_fields`/null
  message fields that some providers reject on multi-turn.
- **Per-run evaluation limits** (identical for every model, and identical across the
  original and resumed portions of the sweep ‚Äî verified from launch commands):
  - **Step limit: 25** LLM turns (`n_calls`, checked before each model query). The
    system prompt mandates exactly one shell command per turn (chained with `&&`/`||`
    if needed), so in practice steps ‚âà shell commands; the loop would run multiple
    commands in a turn if the model emitted multiple tool-calls, but that doesn't occur
    under this prompt. Hitting the limit ends the run as `LimitsExceeded`.
  - **Cost limit: \$0.50** per run (litellm-accounted spend). Hitting it ends the run as
    `LimitsExceeded`.
  - **Per-command timeout: 600 s** (`--timeout`, the default) for each shell command the
    agent issues ‚Äî bounds a single compile/benchmark, not the whole run.
  - **Format-error tolerance: 10** consecutive malformed replies before aborting as
    `RepeatedFormatError` (`--max-format-errors`).
  - A run ends as `Submitted` only when the agent explicitly submits; otherwise it exits
    via one of the limits above and is scored on whatever `solution.py` it left behind
    (so a `LimitsExceeded` run can still score >0 if its last solution was partially
    correct). These exit reasons are tallied in ¬ß7.
- Models benchmarked (cheap, cross-vendor, all verified working via a 2-turn probe):
  **`gpt-5.4-mini`**, **`claude-haiku-4-5`**, **`deepseek-v4-flash`**.
  (Also confirmed working: `qwen3.5-plus`, `glm-5`, `kimi-k2.5`.)

Early observations (perf still filling in):

- **Debug (complete) discriminates cleanly:** `deepseek-v4-flash` **1.00**,
  `claude-haiku-4-5`/`gpt-5.4-mini` **0.93** ‚Äî all three repair planted bugs well; this is

- **Implement separates the field:** `claude-haiku-4-5` (0.68) ‚â≥ `gpt-5.4-mini` (0.57) ‚â´
  `deepseek-v4-flash` (0.14). DeepSeek can *fix* kernels but rarely *authors* one from
  scratch within budget ‚Äî it `LimitsExceeded` on 47/65 runs vs Submitting only 18.
- **Perf is the hardest track and most discriminative:** on the completed subset
  `claude` reaches the repo-tuned target (1.00), `gpt` lands mid (0.78), `deepseek` low
  (0.57) ‚Äî confirming the correctness-gated log-speedup gate is live and far from
  saturated.
- **Track spread** (debug ‚â´ implement ‚â≥ perf, model-dependent) shows the benchmark is
  informative rather than saturated.

## 8. Shortcomings
- **Perf diversity:** the perf track is GEMM-only (fp16/bf16 √ó shapes/detune). Robust and
  discriminative, but narrow; gemv/reduction perf were cut for lack of a clean,
  verifiable speed knob.
- **Debug ‚âÝ historical regression:** bugs are *planted* (contamination-proof, automatic)
  rather than mined from real PRs. The authentic variant (check out a bug-fix commit's
  parent, make its test pass) needs a costly *editable per-task TileLang build*
  (`snapshot_repo.sh` exists; per-task venv/worktree + `ccache` is the missing piece).
- **No clock pinning** (no permission for `nvidia-smi -lgc`); mitigated statistically, but
  perf scores carry small timing noise (rel-std gate + re-measure bound it).
- **Single tilelang version** (0.1.11) and our own `@T.prim_func` template style; newer
  `@tilelang.jit` API surface is untested.
- **Contamination:** TileLang is public; mitigated by private graders, offline agent, and
  (for debug) novel planted bugs, but perfect isolation isn't guaranteed.

## 9. Improvements / future work
- Add a **historical-PR regression track** with per-task editable builds (highest authenticity).
- Broaden perf to **flash-attention / dequant-gemm / conv** with verified tuned targets.
- Add **multi-seed perf repeats** + report score variance/CIs.
- Larger model set and reasoning configs; per-task token/cost accounting in the report.
- **Held-out / mutated specs** to further reduce contamination risk.

    --manifest tasks/manifest.json --gpus 0,1,2,3,4,5,6,7 --step-limit 25 --cost-limit 0.5 \
